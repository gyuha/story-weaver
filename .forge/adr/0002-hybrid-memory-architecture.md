# 하이브리드 메모리 아키텍처 (정형 카드 + 벡터 + 씬-엔티티 링크)

"기억하는 AI"를 순수 벡터 RAG가 아니라, 정형 엔티티 카드(타임라인 상태 포함) + 벡터 검색 + 명시적 씬-엔티티 링크의 하이브리드로 구현한다. 저장소는 PostgreSQL + pgvector로 관계형 데이터와 벡터를 한 DB에 둔다. 순수 벡터 검색은 "관련성"만 줄 뿐 "3화에서 죽은 캐릭터가 10화에 왜 나오나" 같은 상태/사실 추적을 못 하는데, 이 상태 추적이 제품의 핵심 차별점(설정 충돌 감지)이기 때문이다.

## Considered Options

- **순수 벡터 RAG** — 구현이 가장 단순하나 상태·시간 추적이 불가능해 v2 분석 기능(충돌 감지)의 토대가 무너진다.
- **정형 데이터만, 벡터 최소** — 상태 추적은 강하나 작가가 씬-엔티티 링크를 전부 수작업으로 걸어야 해 UX 부담이 크다.
- **하이브리드 (채택)** — 설계 복잡도가 가장 높지만 상태 추적과 의미 검색을 모두 얻고, 차별점이 살아남는다.

## Consequences

- 엔티티 카드 스키마, 타임라인 상태 모델, 임베딩 인덱싱, 링크 추출/제안 로직을 모두 설계·유지해야 한다(가장 무거운 서브시스템).
- pgvector 채택으로 별도 벡터 DB 운영 비용을 없애지만, 대규모 확장 시 전용 벡터 DB 재검토 여지를 남긴다.
