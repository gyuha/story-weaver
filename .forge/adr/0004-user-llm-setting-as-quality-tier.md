# 사용자 LLM 설정 = 품질 티어 선택 (BYOK 배제, 전역 단위)

설정 화면의 LLM 설정에서 사용자가 조작하는 것은 [[품질 티어]](저비용/균형/고품질) **하나**다. 구체 모델명·provider·API 키는 노출하지 않으며(BYOK 아님 — 키는 서버가 보유), 티어는 작품 전체에 전역으로 적용된다. 모델 스위칭(작업 종류별 자동 모델 선택)은 이 티어를 기준으로 그 안에서 동작한다. 현재 백엔드는 서버 전역 환경변수(`LLM_` prefix, `core/config.py`의 `LLMSettings`) 하나로만 LLM을 잡고 있어 "사용자별 LLM 설정"이라는 개념 자체가 없다 — 이 ADR은 그 개념을 어떤 형태로 도입할지의 분기점이다.

## Considered Options

- **품질 티어 선택 + 전역 단위 (채택)** — 소비자 SaaS에 자연스럽고, 키 암호화 저장·과금 전가 같은 부담이 없다. 사용자는 "고품질로 쓸지/저렴하게 쓸지"만 결정하고 구체 모델은 플랫폼이 책임진다. CONTEXT의 모델 스위칭 전략과 매끄럽게 겹친다. UI·데이터 모델이 가장 단순.
- **BYOK (사용자 자기 API 키 입력)** — 파워유저·과금 전가엔 유리하나, 사용자별 키 암호화 저장, provider별 검증, 키 유출 책임, UI 복잡도가 급증한다. MVP에 과하다.
- **작업 종류별 티어 / 구체 모델 직접 선택** — 제어력은 높지만 모델 스위칭이라는 내부 전략을 사용자에게 노출해 초보 작가에게 과하고, 모델 라인업 변경 시 UI 유지보수 부담이 크다.

## Consequences

- 추후 사용자별 LLM 설정을 백엔드에 구현할 때, "티어 → 모델 매핑"을 서버가 소유한다. 사용자 설정 저장 단위는 작품(또는 계정) 1개 enum 필드면 충분하다.
- BYOK 시장을 포기한다. 향후 진입하려면 키 암호화 저장·provider 검증·과금 모델을 별도 ADR로 재설계해야 한다.
- 이번 작업은 mock(프론트 + localStorage) 단계라 위 백엔드 매핑은 비목표다. 티어 enum과 화면 계약만 확정한다.
- 티어 라인업(저비용/균형/고품질)과 각 티어의 실제 모델 매핑은 ADR-0003(상용 LLM 하이브리드)의 모델 스위칭 설계에 종속된다.
